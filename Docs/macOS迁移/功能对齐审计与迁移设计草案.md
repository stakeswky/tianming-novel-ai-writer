# 天命 macOS 版本功能对齐审计与迁移设计草案

日期：2026-05-10

## 目标

将当前 Windows WPF 版天命改造成可在 macOS 上运行的桌面版本，并以原版本功能为对齐基准。

本草案先定义“对齐”的可验收口径、现有项目事实、迁移路线和风险边界。它不是最终实施计划，也不包含代码改动。

## 当前项目事实

- 现有桌面端入口：`Core/App/天命.csproj`
- 当前目标框架：`net8.0-windows10.0.19041.0`
- 当前 UI 框架：WPF，`UseWPF=true`
- 当前应用类型：`WinExe`
- 代码规模快照：
  - XAML 视图：约 155 个
  - C# 文件：约 761 个
  - 主要 UI 文件集中在 `Framework/`，主要业务服务集中在 `Services/`
- 当前本机环境限制：
  - macOS shell 未检测到 `dotnet` 可执行文件
  - 因此尚未完成本机 `dotnet build` 验证

关键外部事实：

- Microsoft WPF 官方文档说明：即使 .NET 是跨平台技术，WPF 只运行在 Windows 上。
- Avalonia 官方文档说明：Avalonia 应用可面向 Windows、macOS、Linux、iOS、Android 和 WebAssembly；桌面端最低 .NET 要求为 .NET 8。

参考：

- https://learn.microsoft.com/en-us/dotnet/desktop/wpf/overview/?view=netdesktop-8.0
- https://docs.avaloniaui.net/docs/supported-platforms

## 功能对齐基准

由于仓库文档已经区分“本地模式”和“服务端模式”，macOS 版本需要分层定义对齐范围。

### A. 核心写作闭环

必须对齐：

- 项目管理
- 五大规则：世界观、角色、势力、地点、剧情
- 创意素材库
- 智能拆书的数据结构与分析流程
- 四层规划：大纲、分卷、章节、蓝图
- 数据中心打包
- 章节生成
- 生成门禁
- 事实快照回写
- 统一校验
- 一致性调和
- 章节修复
- 章节预览
- 版本历史与版本对比
- 打包发布

### B. AI 与对话能力

必须对齐：

- 多模型管理
- Endpoint / Model / API Key 配置
- 多 Key 轮换
- 用量统计
- 提示词管理
- 提示词版本测试
- Agent / Plan / Edit 对话模式
- 会话历史
- 对话压缩
- 项目上下文引用
- 待办执行
- 本地 ONNX 向量搜索
- OpenAI / Anthropic / Google / 兼容接口调用

### C. 编辑器与工作区

必须对齐：

- 三栏工作区
- 左侧导航
- 中央 Markdown 编辑器
- 多标签页
- 仪表盘
- 计划视图
- 主页视图
- 内联编辑
- 右侧 AI 对话面板
- Reference 下拉引用

### D. 框架能力

必须对齐，但允许使用 macOS 原生或跨平台等价实现：

- 主题管理
- 主题导入导出
- AI 配色
- 图片取色
- 系统明暗主题跟随
- 定时主题切换
- 字体管理
- 加载动画
- UI 缩放
- Toast / 应用内通知
- 通知历史
- 勿扰模式
- 声音方案
- 语音播报
- 代理配置
- 日志配置
- 数据清理
- 系统信息
- 系统监控

### E. 用户与服务端能力

需要产品决策后再纳入完整对齐：

- 账号注册与登录
- OAuth 登录
- 账号绑定
- 订阅状态
- 卡密激活
- 服务端公告
- 服务端功能授权
- 强制更新
- 多设备登录状态检查
- SSL Pinning 与保护系统

推荐 macOS v1 采用“本地模式功能全对齐，服务端模式接口预留”的范围。原因是核心写作闭环在本地即可验证，而服务端能力依赖外部服务地址、证书、OAuth ClientId 和授权协议。

## 推荐技术路线

### 方案 1：Avalonia 迁移，复用 C# 服务层

推荐。

做法：

- 新增跨平台解决方案结构。
- 把业务模型、服务、AI、项目数据、生成、校验、追踪逻辑迁到 `net8.0`/`net10.0` 无 UI 类库。
- 新建 Avalonia macOS 桌面壳。
- 将 WPF XAML 逐页迁移为 Avalonia AXAML。
- 替换 Windows 专属服务为平台接口和 macOS 实现。
- 对无法立即等价的服务端/系统集成功能使用能力探测与降级。

优势：

- 保留 C#、MVVM、XAML 思路和大量服务层代码。
- macOS 桌面体验真实，不需要浏览器壳。
- 后续可扩展到 Windows/Linux。

代价：

- 155 个 XAML 视图需要迁移或重建。
- WPF 专属控件、资源、行为、动画和第三方 WPF 包需要替换。
- 初期需要建立平台抽象层，否则 Windows 绑定会反复污染跨平台代码。

### 方案 2：Tauri/Web 前端 + .NET 后端服务

可行但不推荐作为第一路线。

优势：

- macOS 打包成熟。
- UI 可用 Web 技术重做。

代价：

- 现有 XAML 和 WPF 控件几乎不能复用。
- C# ViewModel 与绑定体系要大规模改写。
- .NET 服务与 Web 前端之间需要重新设计 IPC/API 边界。

### 方案 3：Avalonia XPF / 商业兼容层

可作为降本验证路径，但不作为默认开源迁移路线。

优势：

- 理论上可减少 WPF 重写量。

代价：

- 依赖商业授权和兼容层限制。
- 对长期维护、开源协作、深度自定义控件不如原生 Avalonia 清晰。

## 推荐架构

目标结构：

```text
src/
  Tianming.Core/              # 纯模型、通用工具、接口
  Tianming.Application/       # 用例服务、项目数据、生成、校验、追踪
  Tianming.AI/                # AI Provider、SK、Key 轮换、向量搜索
  Tianming.Platform/          # 平台抽象接口
  Tianming.Platform.Mac/      # macOS 实现
  Tianming.Desktop.Avalonia/  # Avalonia UI 壳
tests/
  Tianming.Application.Tests/
  Tianming.AI.Tests/
  Tianming.Platform.Tests/
```

平台抽象优先级：

1. `ISecureStorage`：替代 `ProtectedData`，macOS 使用 Keychain。
2. `INotificationService`：替代 UWP/Windows 通知。
3. `ITrayService`：替代 Windows NotifyIcon。
4. `IAutoStartService`：替代注册表 Run Key。
5. `IFileAssociationService`：替代注册表文件关联。
6. `IUrlProtocolService`：替代注册表协议注册。
7. `IThemeMonitor`：替代 Windows 注册表主题监听。
8. `ISystemInfoService`：替代 WMI/System.Management。
9. `IAudioDeviceService`：替代 NAudio Windows 设备枚举。
10. `ISpeechService`：替代 System.Speech。
11. `IEmbeddedBrowserService`：替代 WebView2.Wpf。
12. `IClipboardService`：隔离 WPF Clipboard。
13. `IAppWindowService`：隔离窗口拖拽、最小化、最大化、闪烁、透明窗口。

## 迁移分期

### M0：可构建基线与功能清单

交付：

- 安装并确认 macOS `dotnet` SDK。
- 记录 Windows 原版构建命令和 macOS 当前失败点。
- 建立功能对齐矩阵。
- 建立 Windows 绑定扫描脚本或检查清单。

验收：

- 能在本机运行 `dotnet --info`。
- 能明确说明当前 WPF 项目为什么不能在 macOS 构建。
- 每个原版功能都有迁移状态：复用、重写、替换、降级、待服务端决策。

### M1：服务层剥离

交付：

- 新增无 UI 类库。
- 迁移项目数据模型、模块服务、生成门禁、校验、追踪、打包发布。
- 去除服务层中的直接 WPF 依赖。

验收：

- 服务层类库可在 macOS 上构建。
- 核心 JSON 数据读写、项目创建、规则 CRUD、章节 CRUD、打包、校验服务有自动化测试。

### M2：AI 与向量能力

交付：

- 迁移 AI Provider 配置。
- 迁移 Key 轮换。
- 迁移 Semantic Kernel 对话服务。
- 迁移 ONNX 向量搜索。
- 建立不依赖真实 API Key 的 fake provider 测试。

验收：

- 本地 fake provider 可完成章节生成闭环。
- 无模型文件时向量搜索可降级到关键词搜索。
- 有模型文件时本地 ONNX 检索可用。

### M3：Avalonia 基础壳

交付：

- 新建 Avalonia 桌面应用。
- 实现主窗口、三栏布局、导航、主题资源、项目选择。
- 接入服务层 DI。

验收：

- macOS 可启动桌面窗口。
- 能创建/打开项目。
- 能展示左侧导航、中央编辑区、右侧对话区。

### M4：核心模块页面迁移

交付：

- 设计模块页面。
- 生成模块页面。
- 校验模块页面。
- AI 模型管理与提示词管理。
- 编辑器、多标签、版本历史、Diff。

验收：

- 按功能矩阵逐项人工验收。
- 关键 ViewModel 行为有测试。
- 章节从规则配置到生成、校验、修复形成最小闭环。

### M5：macOS 平台能力补齐

交付：

- Keychain 安全存储。
- macOS 通知。
- macOS 托盘/菜单栏。
- macOS 打开文件、导入导出、协议处理。
- macOS 系统主题跟随。
- macOS 音频/语音策略。
- macOS 代理配置策略。

验收：

- 每个 Windows 专属功能都有 macOS 等价实现或明确降级说明。
- 不存在未隔离的 Windows API 引用进入 macOS 可构建项目。

### M6：服务端能力接入

交付：

- 登录、注册、OAuth、订阅、卡密、公告、授权心跳。
- SSL Pinning 的跨平台实现。
- 更新检查策略。

验收：

- 需要真实服务端环境与 OAuth 配置。
- 没有服务端配置时，本地模式仍可正常使用。

### M7：打包、签名、公证与回归

交付：

- macOS `.app` 打包。
- Apple Silicon / Intel 架构策略。
- 签名、公证说明。
- 回归测试清单。

验收：

- 干净 macOS 机器可安装并启动。
- 本地写作闭环、AI fake provider 闭环、数据持久化、重启恢复、导入导出通过。

## 当前缺口

- macOS `dotnet` SDK 已定位并用 `global.json` 固定到 `8.0.420`；仍需持续验证打包链路。
- 已建立首个跨平台 ProjectData 类库与测试项目；完整跨平台解决方案仍未建立。
- 已开始剥离 ProjectData 生成门禁/追踪服务层；UI 与大部分平台服务仍未剥离。
- 已建立自动化测试项目并覆盖首批 ProjectData 生成/追踪核心逻辑。
- 尚未有原版 Windows 运行截图或逐项操作基线。
- 服务端能力是否纳入 macOS v1 尚未确认。
