## 1. AI集成服务

**文件位置**: `Services/AIIntegration/`

### AIModelService - AI模型服务

管理AI模型的配置和调用。

```csharp
var modelService = new AIModelService();
var response = await modelService.GenerateAsync(prompt, model);
```

### PromptBuilderService - Prompt构建服务

构建和优化AI提示词。

```csharp
var promptService = new PromptBuilderService();
var prompt = promptService.Build(template, variables);
```

### ModelLibraryService - 模型库服务

管理AI模型库。

```csharp
var libraryService = new ModelLibraryService();
var models = libraryService.GetAvailableModels();
```

### UniversalProvider & 协议适配组件（新）

> ⚠ 当前AI调用链已逐步迁移到新的协议适配架构，所有底层HTTP请求统一由UniversalProvider和协议适配器负责。

- **核心文件位置**
  - `Services/AI/Core/AIService.cs`  
    管理 `AIProvider` / `AIModel` / `UserConfiguration` 等核心数据模型。
  - `Services/AI/Providers/UniversalProvider.cs`  
    通用AI调用入口，根据供应商自动选择协议适配器。
  - `Services/AI/Protocol/`  
    协议适配组件目录，每种“协议族”一个 `*ProtocolAdapter` 实现：
    - `OpenAICompatibleProtocolAdapter` – OpenAI兼容族（OpenAI、DeepSeek、Moonshot、OpenRouter、302ai、siliconflow等）
    - `AnthropicProtocolAdapter` – Claude官方API
    - `GeminiProtocolAdapter` – Google Gemini官方API
    - `AzureOpenAIProtocolAdapter` – Azure OpenAI服务

- **扩展新协议的规范**
  1. **判断是否为全新协议族**：若新厂商提供的是完全不同的REST结构（不是OpenAI兼容，也不是已有几种），需要新增适配器。
  2. 在 `Services/AI/Protocol/` 下新增一个实现 `IAIProtocolAdapter` 的类：
     - `BuildRequest(...)`：构造该协议所需的请求URL、Header和JSON Body；
     - `TryExtractContent(...)`：从该协议的响应JSON中抽取文本内容。
  3. 在 `UniversalProvider.ResolveAdapter(...)` 中增加路由逻辑：
     - 根据 `AIProvider.Id` / `Name` / `ApiEndpoint`（或后续的 `ProtocolType` 字段）选择对应的 `IAIProtocolAdapter`；
  4. 不在UI层暴露“请求格式/响应格式”选择，统一由协议适配层自动处理。

> 后续如需接入新的官方API（例如国内各家原生REST接口），请优先在 `Services/AI/Protocol/` 下新增对应适配器，并只在 `UniversalProvider` 中增加一条路由。

## 7. ModuleServiceBase - 模块服务基类

**文件位置**: `Framework/Common/Services/ModuleServiceBase.cs`  
**命名空间**: `TM.Framework.Common.Services`

### 功能说明

模块服务基类，提供本地分类和数据管理，分类和数据分开存储。

### 基础用法

```csharp
public class OutlineService : ModuleServiceBase<OutlineCategory, OutlineData>
{
    public OutlineService()
        : base(modulePath: "Generate/StrategicOutline",
               categoriesFileName: "categories.json",
               dataFileName: "outlines.json")
    {
    }
    
    // 基类自动提供：
    // - GetAllCategories() - 获取所有分类
    // - GetAllData() - 获取所有数据
    // - AddData() / UpdateData() / DeleteData()
    // - AddCategory() / UpdateCategory() / DeleteCategory()
    
    protected override int OnBeforeDeleteData(string dataId)
    {
        return DataItems.RemoveAll(d => d.Id == dataId);
    }
}
```

### 特性

- ✅ 自动管理本地分类和数据
- ✅ 分类和数据分开存储
- ✅ 完整的CRUD接口
- ✅ 批量启用/禁用
- ✅ 泛型支持，灵活扩展

### 批量启用/禁用

> 用于配合 `DataManagementViewModelBase` 的“一键启用/禁用（BulkToggle）”能力。

**新增方法**：
- `SetCategoriesEnabled(IEnumerable<string> categoryNames, bool enabled)`
- `SetDataEnabledByCategories(IEnumerable<string> categoryNames, bool enabled)`

**约束**：
- `TCategory` 必须实现 `ICategory`，并包含 `IsEnabled` 属性；
- 数据项需具备 `Category`（string）与 `IsEnabled`（bool）属性（基类通过反射访问）。

## 服务层设计原则

### 1. 职责划分

- **Services层**：全局性、跨功能的公共服务
- **Framework层**：功能专属服务

### 2. 使用规范

- 多个模块共享的服务放在Services层
- 单一模块使用的服务放在对应Framework目录

### 3. 依赖注入

建议使用依赖注入管理服务生命周期。
