## 问题1: 主题过渡功能需要点击2次才显示

**问题日期**: 2024-12-19  
**修复状态**: ✅ 已解决  
**影响范围**: Framework/Appearance/Animation/ThemeTransition  
**严重级别**: 高

---

## 📋 问题现象

### 用户反馈
用户点击左侧导航栏的"主题过渡"功能后，**中间内容区域无法立即显示功能界面**，需要**点击第二次**才能看到内容。

### 日志表现
```
[导航] 3级功能点击: ui_settings/animation_settings/theme_transition
[导航] 查询路由配置...
[导航] 找到视图: TM.Framework.Appearance.Animation.ThemeTransition.ThemeTransitionView
[导航] ✓ 视图已创建
[ThemeTransition] 检测到显示器刷新率: 240Hz
[导航] 功能: 主题过渡 🔄
[导航] 添加标签页: ui_settings_animation_settings_theme_transition
[TabbedWorkspace] 标签已添加并选中: ui_settings_animation_settings_theme_transition
[导航] ✓✓✓ 成功
```

**关键观察**：
- 日志显示视图创建成功
- 标签页添加成功
- 但UI界面未能立即渲染显示

---

## 🔍 错误原因分析

### 根本原因
**`ThemeTransitionView.xaml` 缺少资源字典引用**，导致XAML中使用的DynamicResource样式无法在第一次加载时找到。

### 详细说明

#### 1. XAML中使用了大量DynamicResource样式

`ThemeTransitionView.xaml` 使用了以下样式（共13处）：
- `StandardGroupBoxStyle` (4处) - 分组框样式
- `StandardComboBoxStyle` (1处) - 下拉框样式
- `StandardTextBoxStyle` (2处) - 文本框样式
- `SecondaryButtonStyle` (3处) - 次要按钮样式
- `DangerButtonStyle` (1处) - 危险按钮样式
- `PrimaryButtonStyle` (1处) - 主要按钮样式

#### 2. 缺少资源字典引用

**错误的代码结构**：
```xml
<UserControl x:Class="TM.Framework.Appearance.Animation.ThemeTransition.ThemeTransitionView"
             xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
             xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml"
             xmlns:emoji="clr-namespace:Emoji.Wpf;assembly=Emoji.Wpf"
             xmlns:helpers="clr-namespace:TM.Framework.Common.Helpers">
    
    <!-- ❌ 缺少 UserControl.Resources 资源引用 -->
    
    <ScrollViewer VerticalScrollBarVisibility="Auto" HorizontalScrollBarVisibility="Disabled">
        <!-- ... 内容使用了大量 DynamicResource 样式 ... -->
    </ScrollViewer>
</UserControl>
```

#### 3. 为什么第二次点击能正常显示？

**第一次点击**：
1. UserControl实例化
2. XAML解析器尝试查找DynamicResource引用的样式
3. **因为没有引用资源字典，本地查找失败**
4. 尝试从应用程序级别查找
5. 可能因为查找延迟或资源未完全加载，导致控件渲染异常
6. **用户看到空白或不完整的界面**

**第二次点击**：
1. 标签页已存在，直接切换
2. 资源可能已被应用程序级别缓存
3. 或者资源查找机制在第二次尝试时成功
4. **界面正常显示**

## 🔬 问题排查过程

### 1. 初步怀疑 - ViewModel构造函数阻塞（误判）

**假设**：`ThemeTransitionViewModel`构造函数中的`DetectMonitorFPS()`使用WMI查询，可能阻塞UI线程。

**验证**：将FPS检测改为异步执行，但问题依然存在。

**结论**：❌ 不是主要原因

### 2. 对比其他正常工作的功能

**关键发现**：对比`ThemeSelectionView.xaml`（正常工作），发现它包含了资源引用：

```xml
<UserControl.Resources>
    <ResourceDictionary>
        <ResourceDictionary.MergedDictionaries>
            <ResourceDictionary Source="pack://application:,,,/Framework/Common/Styles/AllStyles.xaml"/>
        </ResourceDictionary.MergedDictionaries>
    </ResourceDictionary>
</UserControl.Resources>
```

而`ThemeTransitionView.xaml`**完全缺少**这部分。

### 3. 验证资源引用的必要性

检查`Framework/Common/Styles/AllStyles.xaml`，确认它包含了所有必需的样式：
- ✅ ButtonStyles（StandardButtons.xaml）
- ✅ InputStyles（StandardInputs.xaml）
- ✅ LayoutStyles（StandardLayout.xaml）

**结论**：✅ 找到根本原因

## ✅ 解决方案

### 修改文件
`天命/Framework/Appearance/Animation/ThemeTransition/ThemeTransitionView.xaml`

### 修改内容

在`<UserControl>`标签后，`<ScrollViewer>`标签前，添加资源引用：

```xml
<UserControl x:Class="TM.Framework.Appearance.Animation.ThemeTransition.ThemeTransitionView"
             xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
             xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml"
             xmlns:emoji="clr-namespace:Emoji.Wpf;assembly=Emoji.Wpf"
             xmlns:helpers="clr-namespace:TM.Framework.Common.Helpers">
    
    <!-- ✅ 添加资源字典引用 -->
    <UserControl.Resources>
        <ResourceDictionary>
            <ResourceDictionary.MergedDictionaries>
                <ResourceDictionary Source="pack://application:,,,/Framework/Common/Styles/AllStyles.xaml"/>
            </ResourceDictionary.MergedDictionaries>
        </ResourceDictionary>
    </UserControl.Resources>
    
    <ScrollViewer VerticalScrollBarVisibility="Auto" HorizontalScrollBarVisibility="Disabled">
        <!-- ... 内容 ... -->
    </ScrollViewer>
</UserControl>
```

### 修改说明

1. **Pack URI格式**：`pack://application:,,,/`
   - 这是WPF的Pack URI格式，用于引用应用程序资源
   - 三个逗号表示引用当前程序集的资源

2. **资源路径**：`Framework/Common/Styles/AllStyles.xaml`
   - 相对于项目根目录的路径
   - AllStyles.xaml会自动合并所有标准样式

3. **生效范围**：
   - 该资源字典仅在当前UserControl及其子元素中生效
   - 不影响其他View的资源查找

## 📚 经验教训

### 1. WPF资源查找机制

**DynamicResource查找顺序**：
1. 当前元素的Resources
2. 父元素的Resources（逐级向上）
3. UserControl.Resources
4. Window.Resources
5. Application.Resources

**最佳实践**：
- 使用标准样式的UserControl必须显式引用资源字典
- 不要依赖应用程序级别的资源查找（可能不稳定）

### 2. XAML开发规范

**标准UserControl结构**：
```xml
<UserControl>
    <!-- 1. 资源引用（如果使用DynamicResource） -->
    <UserControl.Resources>
        <ResourceDictionary>
            <ResourceDictionary.MergedDictionaries>
                <ResourceDictionary Source="..."/>
            </ResourceDictionary.MergedDictionaries>
        </ResourceDictionary>
    </UserControl.Resources>
    
    <!-- 2. 内容区域 -->
    <Grid>
        <!-- ... -->
    </Grid>
</UserControl>
```

**检查清单**：
- [ ] 是否使用了DynamicResource样式？
- [ ] 是否引用了包含这些样式的资源字典？
- [ ] 资源路径是否正确（Pack URI格式）？

### 3. 问题排查方法

**有效的排查思路**：

**无效的排查方向**：

## 问题2: ListBox虚拟化导致程序崩溃

**问题日期**: 2024-12-19  
**修复状态**: ✅ 已解决  
**影响范围**: Framework/Common/Styles/ListBoxStyles/StandardListBox.xaml  
**严重级别**: 致命 🔥

---

### 📋 问题现象

#### 错误信息
```
Unhandled exception. System.InvalidOperationException: 
"{DependencyProperty.UnsetValue}"不是属性"Margin"的有效值。
```

#### 崩溃堆栈
```
at System.Windows.DependencyObject.GetEffectiveValue(...)
at System.Windows.FrameworkElement.MeasureCore(Size availableSize)
at System.Windows.Controls.VirtualizingStackPanel.MeasureChild(...)
```

#### 触发条件
1. 导航到"规则配置"或"规则测试"功能
2. ListBox尝试渲染规则列表时
3. 程序立即崩溃退出

#### 日志表现
```
[组件] 4栏布局初始化...
[组件] 4栏布局初始化完成
[主窗口] 初始化...
[主窗口] 初始化完成
.\天命.exe : Unhandled exception. System.InvalidOperationException...
```

**关键观察**：
- 程序在主窗口初始化后立即崩溃
- 错误发生在ListBox的虚拟化测量阶段
- Margin属性获取到了无效值`DependencyProperty.UnsetValue`

---

### 🔍 错误原因分析

#### 根本原因
**在ControlTemplate中使用了 `BasedOn="{StaticResource CardStyle}"` 的样式继承，而CardStyle本身又通过StaticResource引用了其他资源，导致在ListBox虚拟化上下文中资源查找失败。**

#### 详细技术分析

##### 1. 问题代码结构

**错误的样式定义**（`StandardListBox.xaml` 原第18-19行）：

```xml
<Border Padding="15">
    <Border.Style>
        <Style TargetType="Border" BasedOn="{StaticResource CardStyle}">
            <Setter Property="Cursor" Value="Hand"/>
            <Style.Triggers>
                <!-- ... -->
            </Style.Triggers>
        </Style>
    </Border.Style>
    <ContentPresenter/>
</Border>
```

**CardStyle的定义**（`CardStyles/StandardCards.xaml`）：

```xml
<Style x:Key="CardStyle" TargetType="Border">
    <Setter Property="Background" Value="{DynamicResource Surface}"/>
    <Setter Property="BorderBrush" Value="{DynamicResource BorderBrush}"/>
    <Setter Property="BorderThickness" Value="1"/>
    <Setter Property="CornerRadius" Value="{StaticResource MediumRadius}"/>
    <Setter Property="Padding" Value="{StaticResource LargePadding}"/>
    <Setter Property="Margin" Value="{StaticResource MediumMargin}"/>  <!-- ❌ 问题来源 -->
</Style>
```

##### 2. 为什么会失败？

**资源查找链条**：
```
ControlTemplate 
  → Border.Style 
    → BasedOn CardStyle (StaticResource)
      → Margin = {StaticResource MediumMargin}
      → Padding = {StaticResource LargePadding}
      → CornerRadius = {StaticResource MediumRadius}
```

**WPF虚拟化机制**：
1. ListBox使用`VirtualizingStackPanel`进行虚拟化
2. ListBoxItem是延迟创建的，而非一次性全部创建
3. 当ListBoxItem在虚拟化上下文中被创建时：
   - ControlTemplate实例化
   - Border.Style尝试解析`BasedOn="{StaticResource CardStyle}"`
   - CardStyle尝试解析`{StaticResource MediumMargin}`
   - **在虚拟化上下文中，StaticResource的嵌套查找可能失败**
   - 返回`DependencyProperty.UnsetValue`
   - WPF尝试将此无效值赋给Margin属性
   - **抛出InvalidOperationException，程序崩溃**

##### 3. StaticResource vs DynamicResource 在虚拟化中的差异

| 特性 | StaticResource | DynamicResource |
|------|----------------|-----------------|
| 解析时机 | XAML解析时（一次性） | 运行时（每次访问） |
| 性能 | 更快 | 稍慢 |
| 虚拟化兼容性 | ❌ 可能失败 | ✅ 更健壮 |
| 嵌套查找 | ❌ 在ControlTemplate中可能失败 | ✅ 支持 |

**为什么StaticResource在虚拟化中可能失败？**
- StaticResource在XAML编译时解析，依赖资源字典的加载顺序
- 虚拟化上下文中，资源树可能未完全构建
- 嵌套的StaticResource查找链在这种情况下容易断裂

##### 4. 次要问题

**缺失资源定义**：`StandardMargins.xaml`缺少`ExtraLargePadding`：

```xml
<!-- ❌ 缺失此定义 -->
<Thickness x:Key="ExtraLargePadding">30,30,30,30</Thickness>
```

这导致`CardStyles/StandardCards.xaml`第24行的`SpacedCardStyle`引用失败（虽然本次未触发）。

---

### ✅ 解决方案

#### 修改1：修复ListBox样式（主要修改）

**文件**：`天命/Framework/Common/Styles/ListBoxStyles/StandardListBox.xaml`

**修改内容**：

```xml
<!-- ✅ 正确写法：内联定义，避免BasedOn嵌套 -->
<Border Padding="15">
    <Border.Style>
        <Style TargetType="Border">
            <!-- 直接定义所有属性，使用DynamicResource -->
            <Setter Property="Background" Value="{DynamicResource Surface}"/>
            <Setter Property="BorderBrush" Value="{DynamicResource BorderBrush}"/>
            <Setter Property="BorderThickness" Value="1"/>
            <Setter Property="CornerRadius" Value="8"/>
            <Setter Property="Cursor" Value="Hand"/>
            <Style.Triggers>
                <!-- 选中状态：蓝色边框 + 加粗 -->
                <DataTrigger Binding="{Binding RelativeSource={RelativeSource AncestorType=ListBoxItem}, Path=IsSelected}" Value="True">
                    <Setter Property="BorderBrush" Value="{DynamicResource PrimaryColor}"/>
                    <Setter Property="BorderThickness" Value="2"/>
                </DataTrigger>
                <!-- 悬停状态：浅蓝背景 -->
                <Trigger Property="IsMouseOver" Value="True">
                    <Setter Property="Background" Value="{DynamicResource HoverBackground}"/>
                </Trigger>
            </Style.Triggers>
        </Style>
    </Border.Style>
    <ContentPresenter/>
</Border>
```

**修改要点**：
1. ❌ 移除 `BasedOn="{StaticResource CardStyle}"`
2. ✅ 直接内联定义所有样式属性
3. ✅ 使用 `DynamicResource` 替代嵌套的 `StaticResource`
4. ✅ 避免在ControlTemplate中使用样式继承链

**同样修改紧凑型样式**：`CompactCardListBoxItemStyle` 也应用相同的修复。

### 📚 经验教训

#### 1. ControlTemplate中避免BasedOn + StaticResource嵌套

**❌ 危险模式**：
```xml
<ControlTemplate>
    <Border>
        <Border.Style>
            <Style TargetType="Border" BasedOn="{StaticResource SomeStyle}">
                <!-- SomeStyle内部又有StaticResource引用 -->
            </Style>
        </Border.Style>
    </Border>
</ControlTemplate>
```

**✅ 安全模式**：
```xml
<ControlTemplate>
    <Border Background="{DynamicResource Surface}"
            BorderBrush="{DynamicResource BorderBrush}"
            BorderThickness="1"
            CornerRadius="8">
        <!-- 直接设置属性，或使用内联Style + DynamicResource -->
    </Border>
</ControlTemplate>
```

#### 2. 虚拟化控件的资源引用规范

**受影响控件**：
- ListBox / ListView
- DataGrid
- TreeView
- ComboBox (下拉部分)
- 任何使用VirtualizingStackPanel的控件

**最佳实践**：
1. ✅ 优先使用`DynamicResource`而非`StaticResource`
2. ✅ 在ControlTemplate中直接设置属性，避免BasedOn
3. ✅ 如果必须使用Style，确保内联定义，不依赖外部资源
4. ✅ 避免多层资源引用链（Resource A → Resource B → Resource C）

#### 3. StaticResource适用场景

**安全使用StaticResource的场景**：
- Window.Resources / Application.Resources（全局资源）
- 非虚拟化控件的直接属性绑定
- 一次性加载的资源（如几何图形、颜色定义）
- 已确保资源在引用前加载的情况

**必须使用DynamicResource的场景**：
- ✅ 虚拟化控件的ItemContainerStyle
- ✅ ControlTemplate内的资源引用
- ✅ 主题切换需要动态更新的资源
- ✅ 跨ResourceDictionary的引用

#### 4. 资源定义完整性检查

**问题**：引用了不存在的资源导致`DependencyProperty.UnsetValue`

**预防措施**：
1. 创建资源字典时，确保所有引用的资源都已定义
2. 使用命名约定（如`SmallPadding`, `MediumPadding`, `LargePadding`, `XLargePadding`, `ExtraLargePadding`）
3. 定期检查资源引用的完整性
4. 编译时注意警告信息

#### 5. 调试虚拟化问题的方法

**有效排查思路**：
1. ✅ 查看异常堆栈，识别`VirtualizingStackPanel`关键字

### 🔍 错误原因分析

#### 根本原因
**Window使用了StaticResource引用样式（如`StandardGroupBoxStyle`、`SecondaryButtonStyle`等），但Window本身缺少资源字典引用，导致StaticResource无法在Window上下文中找到对应的样式资源。**

#### 详细技术分析

##### 1. 问题代码结构

**错误的Window定义**（原`AvatarUploadDialog.xaml`）：

```xml

        xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
        xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml"
        xmlns:emoji="clr-namespace:Emoji.Wpf;assembly=Emoji.Wpf"
        Title="上传头像" 
        Height="500" Width="600"
        WindowStartupLocation="CenterOwner"
        Background="{DynamicResource ContentBackground}"
        ResizeMode="NoResize">
    
    <!-- ❌ 缺少 Window.Resources 资源引用 -->
    
    <Grid Margin="20">
        <GroupBox Header="预览" Style="{StaticResource StandardGroupBoxStyle}">
            <!-- ❌ StaticResource无法找到样式 -->
        </GroupBox>
    </Grid>
</Window>
```

##### 2. 为什么Window需要显式引用资源？

**Window与UserControl的差异**：

| 特性 | UserControl | Window |
|------|-------------|--------|
| 资源继承 | 自动继承父容器资源 | 独立窗口，无父容器 |
| 资源查找 | 向上查找到Application.Resources | 只查找Window.Resources和Application.Resources |
| StaticResource行为 | 可依赖父级资源 | 必须在Window或Application中定义 |

**Window的资源查找顺序**：
1. Window.Resources（如果未定义，StaticResource立即失败）
2. Application.Resources

**为什么Application.Resources不够？**
- Application.Resources需要在App.xaml中手动合并所有样式字典
- 本项目的标准做法是每个View/Window自行引用所需的资源字典
- 这样更灵活，避免全局资源污染

##### 3. 为什么使用标准窗口（有标题栏）会失败？

**初次尝试**：保留标准Window外观（有标题栏、边框）

问题：
- 即使添加了`Window.Resources`引用
- 标准窗口样式与项目的现代化UI风格不符
- 缺少圆角、阴影等视觉效果
- 不符合项目UI一致性标准

## 问题4: Type.GetType()无法加载自定义类型导致占位符显示

**问题日期**: 2024-12-19  
**修复状态**: ✅ 已解决  
**影响范围**: Services/Navigation/RouteConfig.cs  
**严重级别**: 致命 🔥

---

### 📋 问题现象

#### 用户反馈
用户点击"网络设置 → 代理设置"下的所有子功能（代理设置、代理规则、代理链、代理测试）后，**所有功能都显示"功能开发中..."占位符**，无法正常显示功能界面。

#### 日志表现
```
[导航] 3级功能点击: network_settings/proxy_settings/proxy_setup
[导航] 查询路由配置...
[导航] ⚠ 未找到路由，使用占位视图
[导航] 功能: 代理设置 🌐
[导航] 添加标签页: network_settings_proxy_settings_proxy_setup
[导航] ✓✓✓ 成功
```

**关键观察**：
- 路由ID配置正确（`network_settings/proxy_settings/proxy_setup`）
- 日志显示"未找到路由"，但路由确实已配置
- XAML文件已正确添加资源字典引用
- .csproj文件已正确注册所有View文件
- 所有View的x:Class定义正确

---

### 🔍 错误原因分析

#### 根本原因
**`Type.GetType(className)` 无法加载自定义程序集中的类型，必须提供完整的程序集限定名称（Assembly Qualified Name）。**

#### 详细技术分析

##### 1. 问题代码结构

**错误的类型加载方式**（原`RouteConfig.cs`第236行）：

```csharp
public static Type? GetViewType(string categoryId, string parentId, string leafId)
{
    try
    {
        // 获取类全名
        if (!Routes.TryGetValue(categoryId, out var categoryRoutes))
            return null;
        
        if (!categoryRoutes.TryGetValue(parentId, out var parentRoutes))
            return null;
        
        if (!parentRoutes.TryGetValue(leafId, out var className) || string.IsNullOrEmpty(className))
            return null;
        
        // ❌ 错误：无法加载自定义程序集的类型
        var type = Type.GetType(className);
        
        if (type == null)
        {
            Debug.WriteLine($"[RouteConfig] 未找到类型: {className}");
            return null;
        }
        
        return type;
    }
    catch (Exception ex)
    {
        Debug.WriteLine($"[RouteConfig] 加载视图失败: {ex.Message}");
        return null;
    }
}
```

**Routes字典中的类名格式**：

```csharp
["proxy_settings"] = new()
{
    ["proxy_setup"] = "TM.Framework.Network.Proxy.ProxySetup.ProxySetupView",
    ["proxy_rules"] = "TM.Framework.Network.Proxy.ProxyRules.ProxyRulesView",
    ["proxy_chain"] = "TM.Framework.Network.Proxy.ProxyChain.ProxyChainView",
    ["proxy_test"] = "TM.Framework.Network.Proxy.ProxyTest.ProxyTestView"
}
```

##### 2. Type.GetType()的工作机制

**Type.GetType()只能加载以下类型**：

1. **当前程序集中的类型**（无需程序集名）
2. **mscorlib或其他核心程序集中的类型**（如System.String）
3. **提供了完整程序集限定名的类型**

**完整程序集限定名格式**：

```
命名空间.类名, 程序集名称
```

例如：
```
TM.Framework.Network.Proxy.ProxySetup.ProxySetupView, 天命
```

##### 3. 为什么其他功能正常？

**关键发现**：之前开发的功能（如安全日志、隐私保护）也遇到了同样的问题，但最终通过同样的修复方法解决。

**为什么用户多次遇到相同问题？**
- 修复安全日志时已经解决了此问题
- 但在开发新功能（代理设置）时又忘记应用修复
- **根本原因**：RouteConfig的`GetViewType`方法本身有问题，而不是某个功能的配置问题

##### 4. 为什么添加资源字典无效？

**初步排查思路（错误方向）**：
1. ✅ 检查XAML是否有`UserControl.Resources`引用 → 已添加
2. ✅ 检查是否缺少转换器定义 → 转换器已定义
3. ✅ 检查路由ID是否匹配 → 路由ID正确

### 🔍 错误原因分析

#### 根本原因
**WPF的`<Page>`项会自动处理XAML的代码后置（`.xaml.cs`）并生成中间文件（`.g.cs`），但在某些特定的文件名或目录结构下，MSBuild的文件包含机制会产生冲突，导致同一个文件被编译系统认为重复包含。**

#### 详细技术分析

##### 1. WPF编译机制

**WPF XAML编译流程**：
```
XAML文件 (View.xaml)
  ↓ MSBuild标记为<Page>
  ↓ WPF生成器（MarkupCompilePass1）
  ↓ 生成中间代码
中间文件 (View.g.cs) → obj/Debug/
代码后置 (View.xaml.cs)
  ↓ C#编译器
  ↓ 编译为IL
最终程序集
```

**关键点**：
- `<Page Include="*.xaml">`会触发WPF生成器
- 生成的`.g.cs`文件包含InitializeComponent等方法
- `.xaml.cs`代码后置应该由`<Page>`自动处理，无需显式`<Compile>`

##### 2. 项目中的配置方式

**`.csproj`文件结构**：

```xml
<ItemGroup>
    <!-- SmartAssistant模块 -->
    <Page Include="..\..\Modules\SmartAssistant\**\*.xaml" ...>
      <SubType>Designer</SubType>
      <Generator>MSBuild:Compile</Generator>
    </Page>
    <Compile Include="..\..\Modules\SmartAssistant\**\*.xaml.cs" ... />
    <Compile Include="..\..\Modules\SmartAssistant\**\*ViewModel.cs" ... />
    <Compile Include="..\..\Modules\SmartAssistant\**\Models\*.cs" ... />
</ItemGroup>
```

**问题根源**：
1. `<Page>`已经通过`Generator=MSBuild:Compile`处理了`.xaml`和`.xaml.cs`
2. 但又显式包含了`<Compile Include="**\*.xaml.cs">`
3. MSBuild在某些情况下会认为`.g.cs`文件被重复包含

##### 3. 为什么只有ProtocolHealthCheckView报错？

**尝试的排查**：
- ✅ 检查文件名是否有特殊字符 → 正常
- ✅ 检查命名空间是否正确 → 正确
- ✅ 检查.csproj配置是否与其他模块一致 → 一致
- ✅ 移除显式`.xaml.cs`包含 → 依然报错
- ✅ 排除obj/bin目录 → 依然报错

**最终发现**：
- 问题与文件名本身有关
- MSBuild在处理某些特定名称时会产生路径解析冲突
- 这是MSBuild的已知边缘情况（edge case）

### 🔍 错误原因分析

#### 根本原因
**PasswordBox.Password属性不是DependencyProperty，而是普通的CLR属性，因此不支持WPF的数据绑定机制。这是WPF的设计决策，出于安全考虑。**

#### 详细技术分析

##### 1. 问题代码结构

**错误的XAML写法**：

```xml
<!-- ❌ 错误：直接绑定Password属性 -->
<PasswordBox Password="{Binding SelectedModel.ApiKey, Mode=TwoWay, UpdateSourceTrigger=PropertyChanged}"/>
```

##### 2. 为什么Password属性不支持绑定？

**WPF设计决策**：

| 原因 | 说明 |
|------|------|
| 安全性 | 防止密码在内存中以明文形式存储 |
| 防止内存转储 | 避免密码通过数据绑定机制泄露 |
| 审计跟踪 | 确保密码访问是显式的、可追踪的 |

**技术细节**：
- `PasswordBox.Password`是普通CLR属性，不是DependencyProperty
- WPF的Binding机制只能绑定到DependencyProperty
- 这是WPF框架的强制限制，无法绕过

##### 3. 为什么其他控件可以绑定？

**对比TextBox.Text属性**：

```csharp
// TextBox.Text 是 DependencyProperty（可以绑定）
public static readonly DependencyProperty TextProperty = 
    DependencyProperty.Register("Text", typeof(string), typeof(TextBox), ...);

// PasswordBox.Password 是普通CLR属性（不能绑定）
public string Password { get; set; }
```

---

### ✅ 解决方案

#### 方案：创建PasswordBoxHelper附加属性

**核心思路**：通过附加属性（AttachedProperty）包装PasswordBox，实现双向绑定。

#### 修改步骤

**步骤1：创建PasswordBoxHelper类**

**文件**：`天命/Framework/Common/Helpers/PasswordBoxHelper.cs`

```csharp
using System.Windows;
using System.Windows.Controls;

using TM.Framework.Common.Helpers
{
    /// <summary>
    /// PasswordBox绑定辅助类（PasswordBox.Password不是DependencyProperty，需要通过附加属性实现绑定）
    /// </summary>
    public static class PasswordBoxHelper
    {
        // 是否正在更新标志，防止循环更新
        private static bool _isUpdating = false;

        // 绑定密码的附加属性
        public static readonly DependencyProperty BoundPasswordProperty =
            DependencyProperty.RegisterAttached(
                "BoundPassword",
                typeof(string),
                typeof(PasswordBoxHelper),
                new FrameworkPropertyMetadata(string.Empty, OnBoundPasswordChanged)
                {
                    BindsTwoWayByDefault = true,
                    DefaultUpdateSourceTrigger = System.Windows.Data.UpdateSourceTrigger.PropertyChanged
                });

        // 是否启用绑定的附加属性
        public static readonly DependencyProperty BindPasswordProperty =
            DependencyProperty.RegisterAttached(
                "BindPassword",
                typeof(bool),
                typeof(PasswordBoxHelper),
                new PropertyMetadata(false, OnBindPasswordChanged));

        public static string GetBoundPassword(DependencyObject dp)
        {
            return (string)dp.GetValue(BoundPasswordProperty);
        }

        public static void SetBoundPassword(DependencyObject dp, string value)
        {
            dp.SetValue(BoundPasswordProperty, value);
        }

        public static bool GetBindPassword(DependencyObject dp)
        {
            return (bool)dp.GetValue(BindPasswordProperty);
        }

        public static void SetBindPassword(DependencyObject dp, bool value)
        {
            dp.SetValue(BindPasswordProperty, value);
        }

        // 当BoundPassword属性变化时，更新PasswordBox的Password
        private static void OnBoundPasswordChanged(DependencyObject d, DependencyPropertyChangedEventArgs e)
        {
            if (d is PasswordBox passwordBox && !_isUpdating)
            {
                _isUpdating = true;
                passwordBox.Password = e.NewValue?.ToString() ?? string.Empty;
                _isUpdating = false;
            }
        }

        // 当BindPassword属性变化时，订阅或取消PasswordBox的PasswordChanged事件
        private static void OnBindPasswordChanged(DependencyObject d, DependencyPropertyChangedEventArgs e)
        {
            if (d is PasswordBox passwordBox)
            {
                if ((bool)e.NewValue)
                {
                    passwordBox.PasswordChanged += PasswordBox_PasswordChanged;
                }
                else
                {
                    passwordBox.PasswordChanged -= PasswordBox_PasswordChanged;
                }
            }
        }

        // 当PasswordBox的Password变化时，更新BoundPassword属性
        private static void PasswordBox_PasswordChanged(object sender, RoutedEventArgs e)
        {
            if (sender is PasswordBox passwordBox && !_isUpdating)
            {
                _isUpdating = true;
                SetBoundPassword(passwordBox, passwordBox.Password);
                _isUpdating = false;
            }
        }
    }
}
```

**关键设计**：
1. `BoundPasswordProperty` - 可绑定的附加属性
2. `BindPasswordProperty` - 控制是否启用绑定的开关
3. `_isUpdating` - 防止循环更新的标志
4. 双向同步：`BoundPassword ↔ PasswordBox.Password`

### 📚 经验教训

#### 1. PasswordBox的安全性设计

**教训**：
- PasswordBox.Password不是DependencyProperty是有意为之的设计
- 这是WPF框架的安全考虑，不是bug
- 需要使用附加属性等技术绕过限制

**WPF中其他安全限制**：
- WebBrowser控件的某些属性也不支持绑定
- 某些加密相关的控件也有类似限制

#### 2. 附加属性的强大功能

**AttachedProperty的应用场景**：
1. ✅ 为现有控件添加新功能（如PasswordBoxHelper）
2. ✅ 实现MVVM模式中的行为绑定（如ButtonHelper.ConfirmMessage）
3. ✅ 绕过框架限制（如本次问题）
4. ✅ 跨控件通信（如Panel.ZIndex）

**最佳实践**：
- 使用标志位（`_isUpdating`）防止循环更新
- 在PropertyChanged回调中订阅/取消事件
- 确保双向同步的正确性

#### 3. PasswordBox的替代方案对比

| 方案 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| 附加属性（本方案） | 完全支持MVVM，代码重用性高 | 需要创建Helper类 | ⭐⭐⭐⭐⭐ |
| 代码后置直接访问 | 简单直接 | 破坏MVVM模式 | ⭐⭐ |
| 使用Behavior（MVVM Light） | 优雅，框架支持 | 需要引入第三方库 | ⭐⭐⭐⭐ |
| 自定义PasswordBox控件 | 完全控制 | 实现复杂 | ⭐⭐⭐ |

**推荐理由**：
- ✅ 附加属性是WPF原生机制，不需要第三方库
- ✅ 符合MVVM模式
- ✅ 代码可重用（可用于项目中所有PasswordBox）
- ✅ 性能开销小

#### 4. 防止循环更新的重要性

**问题场景**：
```
PasswordBox.PasswordChanged → 更新BoundPassword 
  → OnBoundPasswordChanged → 更新PasswordBox.Password 
    → PasswordChanged再次触发 → 无限循环 💥
```

**解决方案**：
```csharp
private static bool _isUpdating = false;

private static void OnBoundPasswordChanged(...)
{
    if (!_isUpdating)  // ✅ 检查标志
    {
        _isUpdating = true;
        // 更新操作
        _isUpdating = false;
    }
}
```

{
    private static bool _isUpdating = false;

    public static readonly DependencyProperty BoundPropertyProperty =
        DependencyProperty.RegisterAttached(
            "BoundProperty",
            typeof(string),  // 类型根据需求修改
            typeof(CustomControlHelper),
            new FrameworkPropertyMetadata(default, OnBoundPropertyChanged)
            {
                BindsTwoWayByDefault = true
            });

    public static readonly DependencyProperty BindPropertyProperty =
        DependencyProperty.RegisterAttached(
            "BindProperty",
            typeof(bool),
            typeof(CustomControlHelper),

    // Getter/Setter
    public static string GetBoundProperty(DependencyObject dp) => (string)dp.GetValue(BoundPropertyProperty);
    public static void SetBoundProperty(DependencyObject dp, string value) => dp.SetValue(BoundPropertyProperty, value);
    public static bool GetBindProperty(DependencyObject dp) => (bool)dp.GetValue(BindPropertyProperty);
    public static void SetBindProperty(DependencyObject dp, bool value) => dp.SetValue(BindPropertyProperty, value);

    // 回调实现
    private static void OnBoundPropertyChanged(DependencyObject d, DependencyPropertyChangedEventArgs e)
    {
        if (d is CustomControl control && !_isUpdating)
        {
            _isUpdating = true;
            // 同步到控件
            _isUpdating = false;
        }
    }

    private static void OnBindPropertyChanged(DependencyObject d, DependencyPropertyChangedEventArgs e)
    {
        if (d is CustomControl control)
        {
            if ((bool)e.NewValue)
                control.SomeEvent += Control_EventHandler;
            else
                control.SomeEvent -= Control_EventHandler;
        }
    }

    private static void Control_EventHandler(object sender, EventArgs e)
    {
        if (sender is CustomControl control && !_isUpdating)
        {
            _isUpdating = true;
            SetBoundProperty(control, control.SomeProperty);
            _isUpdating = false;
        }
    }
}

## 🔍 问题分析

### 1. 错误本质
**UserControl 作为 `StandardDialog` 的 `Content` 时，无法访问主题资源。**

### 2. 技术原因
`CustomModelForm.xaml` 是一个独立的 `UserControl`，当被 `StandardDialog.SetContent()` 加载时：
- UserControl 没有自己的资源字典
- 无法访问 `App.xaml` 中的全局资源（如 `TextPrimary`、主题颜色等）
- WPF 尝试解析 `{DynamicResource TextPrimary}` 时找不到资源
- 抛出 `StaticResourceHolder` 异常

### 3. 为什么第4次能成功？
这是WPF资源解析的一个特性：
- 前3次失败后，WPF的资源解析器可能缓存了某些回退值
- 或者延迟加载机制在多次失败后触发了全局资源查找
- **但这是不可靠的行为，不应依赖**

### 4. 对比分析
与 **问题1**、**问题3** 相同的根本原因：
- 问题1: `ThemeTransitionView` 等 UserControl 缺少资源引用
- 问题3: `AddCustomModelDialog` Window 缺少资源引用  
- **问题7**: `CustomModelForm` UserControl 作为Dialog内容缺少资源引用

## 📚 经验总结

### 1. **UserControl 作为 Dialog 内容的规范**
任何 UserControl 如果会被用作：

- 其他动态加载的内容
- 独立的弹窗内容

**必须**在 XAML 中显式引用资源字典：
```xaml
<UserControl.Resources>
    <ResourceDictionary>
        <ResourceDictionary.MergedDictionaries>
            <ResourceDictionary Source="pack://application:,,,/Framework/Common/Styles/AllStyles.xaml"/>
        </ResourceDictionary.MergedDictionaries>
    </ResourceDictionary>
</UserControl.Resources>
```

### 2. **检查清单**
创建新的 UserControl 表单时：
- [ ] 添加资源字典引用
- [ ] 使用 `DynamicResource` 而非 `StaticResource`（如需主题切换支持）
- [ ] 测试在 Dialog 中显示是否正常
- [ ] 测试主题切换是否生效

### 3. **通用检测方法**
如果遇到 `StaticResourceHolder` 异常：
1. 检查 XAML 文件是否有 `<UserControl.Resources>` 或 `<Window.Resources>`
2. 确认是否引用了 `AllStyles.xaml`
3. 如果是动态加载的内容，必须添加资源引用

### 4. **"多次点击才成功"的启示**
这种"随机成功"的行为**是资源未正确引用的典型症状**：
- 不要依赖WPF的回退机制
- 不要期望"多试几次就好了"
- 必须从根本上解决资源引用问题

## 问题8: 多行TextBox文本垂直居中且被截断

**问题日期**: 2025-10-25  
**修复状态**: ✅ 已解决  
**影响范围**: Modules/SmartAssistant/PromptManagement/PromptEditor/PromptEditorView.xaml  
**严重级别**: 中

---

### 📋 问题现象

#### 用户反馈
在"提示词编辑器"界面的"模板内容"输入框中，输入的文本垂直居中显示，而不是从顶部开始。当输入多行文本后，超出部分的文本会被截断，无法完整显示。

#### 技术表现
- `TextBox` 控件的 `VerticalContentAlignment` 属性被设置为 `Top`，但没有生效。
- `TextBox` 应用了 `StandardTextBoxStyle` 样式。
- 多行文本（通过 `AcceptsReturn="True"` 和 `TextWrapping="Wrap"` 启用）无法正常从顶部对齐。

---

### 🔍 错误原因分析

#### 根本原因
**`StandardTextBoxStyle` 的控件模板（ControlTemplate）内部硬编码了内容的垂直对齐方式，导致外部对 `VerticalContentAlignment` 属性的设置被覆盖。**

#### 详细技术分析

##### 1. 问题代码结构

**视图XAML** (`PromptEditorView.xaml`):
```xml
<!-- 尝试设置VerticalContentAlignment="Top"，但无效 -->
<TextBox Text="{Binding TemplateContent}"
         AcceptsReturn="True"
         TextWrapping="Wrap"
         VerticalContentAlignment="Top"
         Style="{StaticResource StandardTextBoxStyle}"/>
```

**样式定义** (`StandardInputs.xaml`):
在 `StandardTextBoxStyle` 的 `ControlTemplate` 中，`ScrollViewer` (即 `PART_ContentHost`) 的 `VerticalAlignment` 被硬编码为 `Center`。

```xml
<Style x:Key="StandardTextBoxStyle" TargetType="TextBox">
    <!-- ... 其他Setter ... -->
    <Setter Property="Template">
        <Setter.Value>
            <ControlTemplate TargetType="TextBox">
                <Border ...>
                    <!-- ❌ 错误：硬编码了垂直对齐方式 -->
                    <ScrollViewer x:Name="PART_ContentHost"
                                  VerticalAlignment="Center"/>
                </Border>
            </ControlTemplate>
        </Setter.Value>
    </Setter>
</Style>
```

##### 2. WPF属性优先级

WPF的依赖属性系统有一套复杂的优先级规则。在这个案例中：

1.  **控件模板内的属性值** (`VerticalAlignment="Center"`) 拥有**极高**的优先级。
2.  **在控件实例上直接设置的属性值** (`VerticalContentAlignment="Top"`) 优先级较低。

因此，模板内的 `Center` 设置覆盖了实例上的 `Top` 设置，导致问题发生。

---

### ✅ 解决方案

#### 修改1：使控件模板支持外部属性（主要修改）

**文件**：`天命/Framework/Common/Styles/InputStyles/StandardInputs.xaml`

**修改内容**：
将 `ScrollViewer` 的 `VerticalAlignment` 属性与 `TextBox` 的 `VerticalContentAlignment` 属性通过 `TemplateBinding` 关联起来。

```xml
<!-- ✅ 正确写法：使用TemplateBinding将模板内部与外部属性关联 -->
<ScrollViewer x:Name="PART_ContentHost"
              VerticalAlignment="{TemplateBinding VerticalContentAlignment}"
              Margin="0"/>
```

这样，控件模板就会响应外部对 `VerticalContentAlignment` 的设置。

#### 修改2：在视图中正确应用样式和属性

**文件**：`天命/Modules/SmartAssistant/PromptManagement/PromptEditor/PromptEditorView.xaml`

**修改内容**：
确保多行 `TextBox` 应用了正确的样式，并覆盖了默认的垂直居中设置。

```xml
<TextBox Style="{StaticResource StandardTextBoxStyle}" 
         Text="{Binding TemplateContent, UpdateSourceTrigger=PropertyChanged}"
         TextWrapping="Wrap" 
         AcceptsReturn="True" 
         BorderThickness="0" 
         Background="Transparent" 
         Foreground="{DynamicResource TextPrimary}"
         VerticalContentAlignment="Top"  <!-- ✅ 必须显式设置为Top -->
         VerticalScrollBarVisibility="Auto" 
         FontFamily="Consolas" 
         FontSize="13" 
         Height="Auto"/> <!-- ✅ 确保高度自适应 -->
```

**修改要点**:
1.  **`Style="{StaticResource StandardTextBoxStyle}"`**: 应用修复后的标准样式。
2.  **`VerticalContentAlignment="Top"`**: 覆盖样式中 `Setter` 设置的默认值 `Center`。
3.  **`Height="Auto"`**: 移除样式中固定的 `Height="36"` 的影响，允许 `TextBox` 随内容增多而扩展高度。

---

### 📚 经验教训

#### 1. 控件模板（ControlTemplate）的属性覆盖问题

**教训**:
在自定义控件模板时，应避免硬编码布局相关的属性（如 `VerticalAlignment`, `HorizontalAlignment`, `Margin` 等）。否则，会使控件失去灵活性，外部设置将不再生效。

**最佳实践**:
- ✅ **使用 `TemplateBinding`**: 将模板内部元素的属性与控件自身的属性关联起来。这是最推荐、性能最好的方式。
- ✅ **示例**: `VerticalAlignment="{TemplateBinding VerticalContentAlignment}"`
- ✅ 这样可以确保控件的使用者能够通过设置控件的公共属性来自定义其外观和行为。

#### 2. 多行TextBox的样式设计

**设计要点**:
- 专门为多行文本创建一个独立的样式（如 `MultiLineTextBoxStyle`），或者确保标准样式足够灵活。
- 多行 `TextBox` 的样式通常需要：
    - `AcceptsReturn="True"`
    - `TextWrapping="Wrap"`
    - `VerticalContentAlignment="Top"`
    - `VerticalScrollBarVisibility="Auto"`
    - `Height` 不应是固定值，应为 `Auto` 或通过 `MinHeight`/`MaxHeight` 控制。

#### 3. 调试WPF样式问题的思路

**有效的排查方法**:
1.  ✅ **检查直接属性设置**: 确认是否在控件实例上正确设置了属性。
2.  ✅ **检查应用的Style**: 查看 `Style` 中是否有 `Setter` 覆盖了你的设置。
3.  ✅ **深入检查ControlTemplate**: 如果前两步无效，问题几乎肯定出在 `ControlTemplate` 内部。使用Visual Studio的"编辑模板"功能或直接查看XAML源码，检查内部元素的属性是否被硬编码。
4.  ✅ **使用实时可视化树 (Live Visual Tree)**: 在调试时，使用此工具可以检查运行时元素的属性值及其来源（本地值、样式、模板等），能快速定位问题。

### 📚 经验教训

#### 1. 隐式样式的资源依赖完整性

**教训**:
- 隐式样式通过`BasedOn`引用其他样式时,**必须**确保被引用的样式资源已加载
- `MergedDictionaries`的加载顺序很重要:被依赖的资源字典要先加载
- 缺少资源会在**运行时动态场景**中暴露,而不是编译时

**最佳实践**:
- ✅ 在`ImplicitStyles.xaml`的`MergedDictionaries`中加载所有需要的基础样式
- ✅ 检查所有`BasedOn="{StaticResource ...}"`的引用是否都有对应的资源定义
- ✅ 使用`DynamicResource`可以提供更好的容错性(但性能略差)

#### 2. ItemsControl动态刷新与样式系统

**发现**:
- ItemsControl刷新时会完全重建内部控件树
- 重建过程中,WPF会严格应用隐式样式
- 静态初始化时的某些回退机制在动态刷新时不可用

**应对策略**:
1. ✅ 确保隐式样式的资源依赖完整
2. ✅ 在View中尽量避免显式样式引用(让隐式样式自动应用)
3. ✅ 测试时要覆盖动态数据绑定和ItemsSource变化的场景

#### 3. StaticResourceHolder异常的排查思路

**有效的排查方法**:
1. ✅ **查看错误行号** - 准确定位到`ImplicitStyles.xaml`的具体行
2. ✅ **检查BasedOn引用** - 确认被引用的资源是否存在
3. ✅ **验证MergedDictionaries** - 检查资源字典是否已加载
4. ✅ **追踪调用堆栈** - 识别是哪个操作触发了样式应用(本例是ItemsControl.OnRefresh)

**无效的排查方向**:
1. ❌ 过度关注View的XAML配置(问题在全局样式)
2. ❌ 删除显式样式引用(隐式样式仍会应用)
3. ❌ 怀疑数据绑定问题(问题在样式资源)

#### 4. 隐式样式配置文件的维护规范

**问题**:
- `ImplicitStyles.xaml`定义了ScrollBar/ScrollViewer的隐式样式
- 但忘记加载包含样式定义的`StandardScrollbar.xaml`
- 这种配置不一致容易在动态场景中暴露

**改进措施**:
1. **创建检查清单**:
   ```markdown
   ImplicitStyles.xaml 配置检查:
   - [ ] Button隐式样式 → StandardButtons.xaml已加载
   - [ ] TextBox隐式样式 → StandardInputs.xaml已加载
   - [ ] ComboBox隐式样式 → StandardInputs.xaml已加载
   - [ ] ListBox隐式样式 → StandardListBox.xaml已加载
   - [ ] ScrollBar隐式样式 → StandardScrollbar.xaml已加载 ✅ 本次修复
   - [ ] ScrollViewer隐式样式 → StandardScrollbar.xaml已加载 ✅ 本次修复
   ```

2. **自动化验证**:
   - 编写单元测试,确保所有`BasedOn`引用的资源都存在
   - 使用XAML静态分析工具检测资源引用完整性

3. **Code Review重点**:
   - 修改`ImplicitStyles.xaml`时,必须验证资源依赖
   - 添加新的隐式样式时,确保对应的资源字典已加载

#### 5. 资源字典加载顺序的重要性

**加载顺序规则**:
```xml
<ResourceDictionary.MergedDictionaries>
    <!-- 1. 基础资源(Margins, Geometry) -->
    <ResourceDictionary Source="StandardMargins.xaml"/>
    <ResourceDictionary Source="StandardGeometry.xaml"/>
    
    <!-- 2. 具名样式(可被隐式样式BasedOn引用) -->
    <ResourceDictionary Source="ButtonStyles/StandardButtons.xaml"/>
    <ResourceDictionary Source="InputStyles/StandardInputs.xaml"/>
    <ResourceDictionary Source="ScrollbarStyles/StandardScrollbar.xaml"/>
    
    <!-- 3. 复合样式(可能依赖前面的样式) -->
    <ResourceDictionary Source="CardStyles/StandardCards.xaml"/>
    <ResourceDictionary Source="ListBoxStyles/StandardListBox.xaml"/>
</ResourceDictionary.MergedDictionaries>
```

**原则**:
- ✅ 被依赖的资源要先加载
- ✅ 基础资源 → 具名样式 → 复合样式 → 隐式样式
- ✅ 如果不确定顺序,使用`DynamicResource`替代`StaticResource`

## 问题10: 创作素材分类节点点击无响应

**问题日期**: 2025-11-13  
**修复状态**: ✅ 已解决  
**影响范围**: Modules/Creation/Materials/CreativeMaterial  
**严重级别**: 中

---

### 📋 问题现象

#### 用户反馈
- 参考资料模块的分类选择正常显示，创作素材模块在点击分类节点后右侧表单不更新，仍停留在空白状态。
- PowerShell 调试输出显示 `TreeNodeItem` 的 `Tag` 为 `KnowledgeCategory` 时没有进一步的表单日志。

#### 操作复现
1. 打开“创作素材”界面。
2. 在分类树中点击任意分类节点（例如“世界观分类/世界观分类”）。
3. 右侧“素材详情”面板保持空白，没有加载对应分类信息或新建表单。

---

### 🔍 根本原因

`CreativeMaterialViewModel.SelectBlock(TreeNodeItem? node)` 方法仅对 `KnowledgeBlock` 节点进行处理：

```csharp
if (node?.Tag is not KnowledgeBlock block) return;

foreach (var rootNode in TreeData)
{
    ClearNodeSelection(rootNode);
}

node.IsSelected = true;
SelectedKnowledgeBlock = block;
```

当用户点击分类节点（`KnowledgeCategory`）时，方法直接返回，不会更新 `SelectedKnowledgeBlock`，导致表单绑定对象为空且界面无变化。

---

### ✅ 解决方案

#### 代码修改
文件：`Modules/Creation/Materials/CreativeMaterial/CreativeMaterialViewModel.cs`

更新 `SelectBlock` 方法，区分数据节点与分类节点：

```csharp
private void SelectBlock(TreeNodeItem? node)
{
    if (node == null)
    {
        return;
    }

    foreach (var rootNode in TreeData)
    {
        ClearNodeSelection(rootNode);
    }

    node.IsSelected = true;

    if (node.Tag is KnowledgeBlock block)
    {
        SelectedKnowledgeBlock = block;
        TM.App.Log($"[KnowledgeBlockManagement] 选择知识块: {block.Title}");
    }
    else if (node.Tag is KnowledgeCategory category)
    {
        TM.App.Log($"[KnowledgeBlockManagement] 选择分类: {category.Name}");
        CreateAndSelectNewBlock(category.Name, showToast: false);
    }
}
```

核心思路：
- 点击分类节点时，清空当前选择并调用 `CreateAndSelectNewBlock` 根据分类预置新素材，表单随即绑定到新建对象。
- 点击数据节点时，保持原有逻辑，仍然加载已存在的素材。

#### 相关文件
- `CreativeMaterialView.xaml`: 右侧表单绑定到 `SelectedKnowledgeBlock`，无需改动。
- `CreativeMaterialViewModel.cs`: `SelectBlock` 与 `CreateAndSelectNewBlock` 共同作用，确保分类与数据节点都能触发表单更新。
